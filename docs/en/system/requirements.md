# 4.2.3. System Software Security Requirements

These are security requirements to protect the system software that controls the core operations of the robot, such as the OS, kernel, boot chain, and firmware, responding to the system software threats identified in Chapter 3. The detailed security requirements are as follows.

| Security Requirement | Mitigation Measure | Checklist ID (Vulnerability Check) | Security Solution |
| :--- | :--- | :--- | :--- |
| **Principle of Least Privilege** | Protect lateral movement paths between accounts/services with a least privilege configuration. | `[IA-01]` User identification and authentication<br>`[IA-09]` User account management<br>`[IA-10]` User identifier management<br>`[UC-01]` User authorization | Server Access Control Solution |
| **Session Management** | Protect against session abuse with session locking, termination, and concurrent session control. | `[UC-03]` Session lock<br>`[UC-04]` Remote session termination<br>`[UC-05]` Concurrent session control | |
| **Intrusion Detection** | Protect against abnormal login and movement activities with real-time intrusion detection. | `[ER-02]` Continuous monitoring<br>`[ER-03]` Security event notification | Industrial IDS |
| **Regular Security Checks & Hardening** | Protect against vulnerability-based attacks with regular scanning, security hardening, and disabling unnecessary services. | `[SI-05]` Input validation<br>`[SI-07]` Error handling<br>`[RA-07]` Security configuration management<br>`[RA-08]` Restriction of unneeded functionality, ports, and services<br>`[CR-03]` Documentation and testing of recovery procedures for cyber incidents<br>`[CR-04]` Definition of RTO and RPO<br>`[CR-08]` Establishment of official procedures for reporting and handling security vulnerabilities<br>`[CR-10]` Procedure for improving security measures based on incidents/vulnerabilities<br>`[WS-20]` Minimization of network interface exposure | Source Code Analysis Tool,<br>Industrial EDR,<br>Security Certification Consulting |
| **Component Management** | Protect against vulnerabilities in open-source and external modules with SBOM-based updates and component management. | `[SC-01]` Manage open source software components<br>`[SC-02]` Manage EoS<br>`[SC-03]` Establish and apply vendor security capability assessment criteria<br>`[SC-04]` Prepare for security impact assessment and response to supply chain changes<br>`[SC-05]` Manage component vulnerabilities<br>`[SC-07]` Implement rapid patch mechanism<br>`[WS-19]` Maintenance of up-to-date software and hardware | Source Code Analysis Tool,<br>Patch Management Solution |
| **Boot Setting Protection** | Protect against tampering of boot images and initial execution code with an integrity verification mechanism, and protect against boot parameter manipulation with configuration locking. | `[UC-11]` Control of physical diagnostic and test interfaces<br>`[SI-04]` Integrity of software and data<br>`[SI-11]` Physical tamper resistance and detection<br>`[SC-06]` Verify integrity of robot firmware and software updates<br>`[SI-04]` Integrity of software and data | Board with TPM |
| **Update Verification** | Protect against tampered update packages with signature-based integrity verification and log generation. | `[SC-06]` Verify integrity of robot firmware and software updates<br>`[WS-13]` Wireless network update integrity and authenticity verification<br>`[WS-17]` Log generation<br>`[WS-18]` Inclusion of time information | Patch Management Solution |
| **Safe Rollback** | Protect against patch failures with an automatic rollback and recovery function. | `[CR-06]` Rollback or recovery function in case of update failure | Patch Management Solution |
| **Security Update Policy** | Protect against improper update management with a regular patch provision and application procedure. | `[SI-10]` Update support verification<br>`[CR-05]` Timely provision and application of security updates and patches<br>`[WS-12]` Update mechanism provision | Patch Management Solution |
| **Kernel Hardening** | Protect against kernel memory attacks with hardening techniques like memory protection and ASLR. | `[SI-02]` Malware protection<br>`[SC-05]` Manage component vulnerabilities | |
| **Allow Only Signed Drivers** | Protect against the execution of malicious drivers/kernel modules with signature verification-based module restriction. | `[SI-04]` Integrity of software and data<br>`[WS-23]` Equipment integrity assurance | |
| **Process Integrity Verification** | Protect against runtime code hooking and tampering with process integrity verification. | `[SI-03]` Security function verification<br>`[SI-04]` Integrity of software and data<br>`[ER-02]` Continuous monitoring | Industrial EDR |
| **Event Analysis** | Protect against missed security events with regular log analysis. | `[CR-09]` Securing sufficient logs and supporting security event analysis | |
| **Audit Log Protection Management** | Protect against log tampering/deletion with access control, encryption, and rotation policies. | `[SI-09]` Protection of audit-related information<br>`[UC-06]` Audit log generation<br>`[UC-07]` Audit log storage capacity management<br>`[UC-08]` Response to audit processing failures<br>`[UC-09]` Time control<br>`[CR-09]` Securing sufficient logs and supporting security event analysis<br>`[WS-18]` Inclusion of time information<br>`[WS-23]` Equipment integrity assurance<br>`[ER-01]` Audit log access management | Backup Management System |
| **Diagnostic Function Access Control** | Protect against the exploitation of debug/diagnostic interfaces with authentication-based access control. | `[IA-02]` Device identification and authentication<br>`[SI-11]` Physical tamper resistance and detection<br>`[UC-01]` User authorization | |
