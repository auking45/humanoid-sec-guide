# 4.1.11. Wireless Security Checklist

The Wireless Security Checklist is for protecting the wireless communication channels used by the robot and meeting the minimum security requirements for wireless equipment. It includes wireless authentication, encryption, and wireless protocol protection, aiming to prevent eavesdropping and forgery of wireless communications.

| ID | Checklist Item | Description |
| :--- | :--- | :--- |
| WS-01 | Wireless User Identification<br>and Authentication | Authentication and access management must be performed for users and terminal levels accessing the robot system's wireless interface, as well as for all user levels, using an appropriate authentication mechanism. |
| WS-02 | Authenticator Validity<br>Verification | The authentication mechanism must verify the relevant properties of all authentication means used, based on information available in the operating environment. |
| WS-03 | Authenticator Change | The authentication mechanism must allow users to change the authentication means they use (e.g., passwords, tokens), as long as it does not conflict with security objectives. |
| WS-04 | Password Strength | All set passwords must be sufficiently secure and unique. Default passwords must be forced to be changed by the user or managed securely. All passwords must be set in compliance with the strength requirements of international standards such as NIST SP 800-63B. |
| WS-05 | Brute-force Protection | The authentication mechanism must be designed to withstand brute-force attacks. That is, appropriate defense measures must be applied to prevent an attacker from discovering a password through repeated authentication attempts or causing a Denial of Service (DoS) state. |
| WS-06 | Asset Access Control | Access to the basic information or assets of all users participating in wireless communication must be controlled so that only authorized users can access them. |
| WS-07 | Secure Data Protection | A secure storage mechanism must be provided to safely store and manage sensitive information and personal data within the network, as well as data at risk of forgery or alteration. |
| WS-08 | Stored Data Integrity<br>Verification | The equipment must provide a function to protect the integrity of stored security assets and personal information assets to prevent and detect data tampering. |
| WS-09 | Wireless Communication Data<br>Protection | The confidentiality of data transmitted and received over the wireless network must be protected by applying security measures such as encryption to prevent external exposure, leakage, or eavesdropping. |
| WS-10 | Wireless Communication Data<br>Integrity Verification | An integrity verification mechanism, such as hash value verification or digital signature, must be applied to ensure that data transmitted over the wireless network has not been altered during transmission. |
| WS-11 | Wireless Communication<br>Replay Attack Prevention | Data transmitted over the wireless network must be protected from being exploited in replay attacks. |
| WS-12 | Update Mechanism Provision | An update function must be provided for software that affects security and privacy assets. Especially when connected to the internet, updates should be possible in various ways, such as automatically or based on approval. |
| WS-13 | Wireless Network Update<br>Integrity and Authenticity<br>Verification | The integrity of software and firmware updated automatically or manually during a remote update via a wireless network must be verified. |
| WS-14 | Cryptographic Adequacy | The cryptographic technology used to protect security assets or network assets must follow the latest security best practices. In addition, the purpose and expected lifespan of each cryptographic use must be clearly documented for systematic cryptographic policy management. |
| WS-15 | Appropriate Cryptographic<br>Key Usage | All confidential cryptographic keys used must have a security strength of at least 112 bits, and the latest security best practices must be followed when generating cryptographic keys. |
| WS-16 | Cryptographic Key Uniqueness | Pre-installed confidential cryptographic keys must be managed to be practically unique per device. However, if a shared key is required for initial trust establishment or to perform a specific function, the purpose and security measures must be clearly documented. |
| WS-17 | Log Generation | Logs must be recorded for internal security activities and events related to major assets. |
| WS-18 | Inclusion of Time Information | All log data stored in the equipment's permanent storage must include time-related information such as a timestamp. |
| WS-19 | Maintenance of Up-to-date<br>Software and Hardware | The product or system must maintain its software and hardware in an up-to-date state to ensure that there are no publicly known exploitable vulnerabilities. Updates and patches must be applied regularly. Components with known vulnerabilities must be addressed immediately, and patch priority should be set according to the risk level. |
| WS-20 | Minimization of Network<br>Interface Exposure | The product or system must minimize the services exposed through the network interface. The interface configuration related to optional services in use must be clearly set, and the exposed list must be documented and managed. |
| WS-21 | Restriction of External<br>Physical Interfaces | Unnecessary external physical interfaces in the product or system must be removed. |
| WS-22 | External Input Management | The system must perform appropriate validation on external inputs and be designed to protect the system from them. If external detection sensors or detection functions are included, their operating principles, range, and linkage methods must be clearly documented to provide sufficient information. |
| WS-23 | Equipment Integrity Assurance | The equipment must be designed to maintain its integrity from physical damage, unauthorized changes, component replacement, etc. |
