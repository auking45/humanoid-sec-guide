# 네트워크 휴머노이드 특화 보안 요구사항

본 문서는 로봇과 클라우드 VLM 간의 데이터 전송 기밀성 유지와 제어 명령의 무결성 보장, 그리고 로봇 내부(이기종 보드 간) 통신 보호를 위한 통신 보안 요구사항을 정의합니다. 기존 KISA 보안 요구사항을 보완하며, 세부 휴머노이드 특화 보안 요구사항은 다음과 같습니다.

| ID | 분류 (Category) | 보안 요구사항 (Item) | 상세 요구사항 (Detailed Requirements) |
| :--- | :--- | :--- | :--- |
| **COM-01** | 인증 및 암호화 | **Mutual TLS (mTLS)** | TLS 1.3을 기반으로 로봇과 서버가 상호 인증을 수행해야 함. |
| **COM-02** | 인증 및 암호화 | **HW Key Binding** | 클라우드 접속용 개인키는 TPM 2.0 또는 TEE 내부에 보관되어 유출이 불가능해야 함. |
| **COM-03** | 인증 및 암호화 | **E2E Encryption** | 모든 영상 데이터 및 제어 명령은 종단간 암호화되어야 함. |
| **COM-04** | 인증 및 암호화 | **Forward Secrecy** | 세션 키 탈취 시 과거 데이터 보호를 위해 PFS 지원 사이퍼 스위트를 사용해야 함. |
| **COM-05** | 명령 무결성 및 권한 제어 | **Command Signing** | 클라우드 제어 명령은 서버 HSM에서 디지털 서명되어야 하며 로봇은 이를 검증함. |
| **COM-06** | 명령 무결성 및 권한 제어 | **Replay Protection** | Nonce와 Timestamp를 사용하여 캡처된 명령의 재전송 공격을 차단해야 함. |
| **COM-07** | 명령 무결성 및 권한 제어 | **RBAC Enforcement** | 정보 조회와 물리적 구동(모터 제어) 권한을 엄격히 계층화하여 분리함. |
| **COM-08** | 명령 무결성 및 권한 제어 | **Command Expiry** | 제어 명령은 짧은 유효 시간(TTL)을 가져야 하며 만료 시 즉시 폐기함. |
| **COM-09** | 네트워크 가용성 및 성능 | **Secure Tunneling** | WireGuard 또는 전용 VPN을 통해 외부 노출을 차단한 터널링을 구성함. |
| **COM-10** | 네트워크 가용성 및 성능 | **QUIC Optimization** | 실시간성 확보를 위해 TCP 대신 UDP 기반 QUIC(HTTP/3) 프로토콜 사용을 권장함. |
| **COM-11** | 네트워크 가용성 및 성능 | **Rate Limiting** | 초당 명령 수신 횟수를 제한하여 DDoS 및 오작동 명령 폭주를 방어함. |
| **COM-12** | 네트워크 가용성 및 성능 | **Fail-Safe Mode** | 통신 단절(Heartbeat Timeout) 시 로봇은 즉시 사전 정의된 안전 정지 모드로 전환함. |
| **COM-13** | 데이터 프라이버시 | **Data Masking** | 클라우드 전송 전 로컬에서 얼굴 등 민감 정보에 대한 비식별화 처리를 수행함. |
| **COM-14** | 데이터 프라이버시 | **Sensor Provenance** | 전송되는 영상 프레임에 TEE에서 생성한 디지털 워터마크(서명)를 포함함. |
| **LOC-01** | 보드 간 통신 인증 및 보안 | **Internal mTLS/Auth** | x86과 NVIDIA 보드 간의 모든 통신은 TLS 1.3 기반의 상호 인증(mTLS) 또는 HMAC 기반의 메시지 인증을 통해 발신처를 상호 검증해야 함. |
| **LOC-02** | 보드 간 통신 인증 및 보안 | **Traffic Segmentation** | VLAN(Virtual LAN) 설정을 통해 고대역폭 영상 데이터(Data Plane)와 저대역폭 핵심 제어 신호(Control Plane)의 네트워크 경로를 논리적으로 엄격히 분리해야 함. |
| **LOC-03** | 보드 간 통신 인증 및 보안 | **Static IP & ARP Binding** | 내부 네트워크의 IP 스푸핑(Spoofing)을 방지하기 위해 고정 IP를 할당하고, 각 보드의 MAC 주소를 ARP 테이블에 정적으로 바인딩하여 비인가 장치의 개입을 차단해야 함. |
| **LOC-04** | 보드 간 통신 인증 및 보안 | **SROS2 Implementation** | 로컬 ROS 2 환경에서 통신 노드 간 암호화, 인증 및 접근 제어 정책(Policy)을 강제하기 위해 SROS2(DDS-Security)를 활성화해야 함. |
| **LOC-05** | 상호 감시 및 하트비트 | **Dual-Watchdog System** | 두 보드 간에 10ms 이하 주기의 하트비트(Heartbeat)를 교환하며, 응답 단절 또는 지연 발생 시 두 보드 모두 즉시 로봇을 안전 정지(Safe Stop) 상태로 전환해야 함. |
| **LOC-06** | 상호 감시 및 하트비트 | **Cross-Attestation** | NVIDIA 보드는 x86 보드의 무결성 상태(예: Audit Log 요약, 커널 서명 값)를 주기적으로 검증하며, 이상 감지 시 즉시 클라우드에 경고를 전송하고 제어권을 회수해야 함. |
| **LOC-07** | 상호 감시 및 하트비트 | **Rate Limiting (Internal)** | 보드 간 통신 대역폭을 물리적 한계치 이하로 제한(Rate Limiting)하여, 특정 노드의 트래픽 폭주(DoS)가 제어 시스템 전체에 영향을 주지 않도록 방어해야 함. |
| **LOC-08** | 인터페이스 및 물리 포트 보안 | **Port Lockdown** | 내부 스위치 및 보드에서 사용하지 않는 모든 물리적/논리적 포트는 비활성화하며, 외부 노출 포트는 인가된 장치만 접속 가능한 화이트리스트 정책을 운영해야 함. |
| **LOC-09** | 인터페이스 및 물리 포트 보안 | **Intrusion Detection (IDS)** | 내부 이더넷 트래픽의 패턴을 분석하여 평소와 다른 비정상적인 프로토콜 호출이나 데이터 요청을 탐지하는 경량 침입 탐지 시스템(IDS)을 운영해야 함. |
| **LOC-10** | 인터페이스 및 물리 포트 보안 | **Secure Key Exchange** | 보드 간 암호화 통신에 필요한 세션 키는 절대 디스크에 저장하지 않으며, 부팅 시 각 보드의 보안 영역(TEE) 간 보안 채널을 통해 동적으로 생성 및 교환해야 함. |
