# 시스템 소프트웨어 휴머노이드 특화 보안 요구사항

본 문서는 산업용 휴머노이드의 하드웨어 신뢰 루트(Root of Trust) 구축과 런타임 시스템 무결성 보장을 위한 기술적 휴머노이드 특화 요구사항을 정의합니다. 기존 KISA 보안 요구사항을 보완하며, 세부 휴머노이드 특화 요구사항은 다음과 같습니다.

| ID | 분류 (Category) | 보안 요구사항 (Item) | 상세 요구사항 (Detailed Requirements) |
| :--- | :--- | :--- | :--- |
| **SYS-01** | 하드웨어 신뢰 루트 및 부팅 보안 | **Hardware Root of Trust** | 모든 보안 연산은 하드웨어(eFuse, TPM 2.0, TEE)에 고정된 유일한 식별자와 키를 기반으로 수행되어야 함. |
| **SYS-02** | 하드웨어 신뢰 루트 및 부팅 보안 | **Chain of Trust Booting** | ROM → Bootloader → Kernel → RootFS로 이어지는 부팅 전 과정에서 이전 단계가 다음 단계의 디지털 서명을 검증해야 함. |
| **SYS-03** | 하드웨어 신뢰 루트 및 부팅 보안 | **Custom Key Usage** | NVIDIA/Intel의 기본 키가 아닌, 개발사가 생성한 고유한 서명 키를 하드웨어에 퓨징(Fusing)하여 사용해야 함. |
| **SYS-04** | 하드웨어 신뢰 루트 및 부팅 보안 | **Anti-Rollback** | 보안 패치가 완료된 최신 펌웨어를 구버전(취약 버전)으로 강제 교체할 수 없도록 하드웨어 보안 카운터를 적용해야 함. |
| **SYS-05** | 하드웨어 신뢰 루트 및 부팅 보안 | **JTAG/Debug Lock** | 양산 및 파트너사 제공 시점에서는 물리적 디버깅 인터페이스(JTAG, UART 등)를 하드웨어적으로 비활성화(Fuse Blow)해야 함. |
| **SYS-06** | 런타임 시스템 보호 | **Read-Only RootFS** | 런타임 시 시스템 영역(/bin, /usr 등)은 읽기 전용으로 마운트되어야 하며, 변경 시도는 하드웨어 예외를 발생시켜야 함. |
| **SYS-07** | 런타임 시스템 보호 | **Kernel Hardening** | KASLR, Stack Protector, CFI 등 최신 커널 보안 옵션을 활성화하여 컴파일 및 배포되어야 함. |
| **SYS-08** | 런타임 시스템 보호 | **Process Isolation** | 핵심 제어 로직과 AI 학습 프로세스는 엄격히 분리된 권한(User/Group)으로 실행하며, 상호 간섭을 차단해야 함. |
| **SYS-09** | 런타임 시스템 보호 | **Resource Limiting** | 특정 프로세스가 자원을 독점하여 시스템을 마비시키지 않도록 cgroups 기반의 자원 할당 제한을 적용해야 함. |
| **SYS-10** | 런타임 시스템 보호 | **Zero-Configuration** | 불필요한 네트워크 서비스, 데몬, 쉘 도구(Compiler 등)를 완전히 제거하여 공격 표면(Attack Surface)을 최소화해야 함. |
| **SYS-11** | 저장 데이터 및 자산 보호 | **Full Disk Encryption** | 로봇 내부의 모든 비휘발성 저장 장치(SSD, eMMC)는 AES-256 이상의 알고리즘으로 암호화되어야 함. |
| **SYS-12** | 저장 데이터 및 자산 보호 | **Binding to Hardware** | 디스크 복호화 키는 TPM/TEE 내부에 저장되어야 하며, Secure Boot가 성공한 상태에서만 메모리로 로드되어야 함. |
| **SYS-13** | 저장 데이터 및 자산 보호 | **IP Protection** | 핵심 바이너리는 스트립(Strip) 처리 및 난독화(Obfuscation)를 거쳐야 하며, TEE 내에서만 복호화되어 실행되는 구조를 지향함. |
| **SYS-14** | 저장 데이터 및 자산 보호 | **Secure Key Lifecycle** | 모든 보안 키(SSH, TLS, API Key)는 파일 시스템에 평문으로 저장되지 않아야 하며, 하드웨어 보안 영역 내에서 관리되어야 함. |
| **SYS-15** | 물리적 침입 탐지 및 대응 | **Chassis Intrusion** | 로봇 외장 케이스 개방 시 기계적 스위치나 센서가 이를 감지하여 보안 위반 이벤트를 즉시 생성해야 함. |
| **SYS-16** | 물리적 침입 탐지 및 대응 | **Self-Destruction (Logic)** | 심각한 물리적 침입 감지 시, TEE 내부에 보관된 핵심 암호화 키를 스스로 파기(Zeroization)하여 데이터 유출을 막아야 함. |
| **SYS-17** | 물리적 침입 탐지 및 대응 | **Port Lockdown** | 외부 노출 포트(USB, LAN)는 인증된 장치 외에는 데이터 통신이 불가능하도록 커널 레벨 화이트리스트 정책을 적용해야 함. |
| **SYS-18** | 커널 자가 보호 및 메모리 보안 | **KASLR 활성화** | 커널 영역의 주소 공간 배치 난독화(KASLR)를 활성화하여 공격자가 특정 커널 함수의 메모리 주소를 예측하지 못하도록 해야 함. |
| **SYS-19** | 커널 자가 보호 및 메모리 보안 | **PXN/PAN 적용** | 권한 실행 방지(PXN) 및 권한 접근 방지(PAN) 기능을 활성화하여 커널 모드에서 사용자 영역의 악성 코드를 실행하거나 데이터를 탈취하는 것을 차단해야 함. |
| **SYS-20** | 커널 자가 보호 및 메모리 보안 | **Stack Protector** | 모든 커널 및 핵심 바이너리 컴파일 시 'Stack Canaries'를 삽입하여 버퍼 오버플로우(Buffer Overflow) 공격을 감지하고 차단해야 함. |
| **SYS-21** | 커널 자가 보호 및 메모리 보안 | **ReadOnly Kernel Data** | 커널의 데이터 영역 중 변경이 필요 없는 부분을 읽기 전용으로 설정하여 런타임 중 커널 데이터 변조를 방지해야 함. |
| **SYS-22** | 접근 제어 및 프로세스 격리 | **AppArmor 프로필 강제** | LSM(Linux Security Module)인 AppArmor를 사용하여 ROS 2 노드, 센서 드라이버 등 각 프로세스가 허용된 파일, 네트워크 포트만 사용하도록 엄격한 프로필을 적용해야 함. |
| **SYS-23** | 접근 제어 및 프로세스 격리 | **최소 권한 원칙 (Least Privilege)** | 로봇의 모든 응용 프로그램은 `root` 권한이 아닌 전용 시스템 계정으로 실행되어야 하며, `sudo` 사용 이력을 엄격히 관리해야 함. |
| **SYS-24** | 접근 제어 및 프로세스 격리 | **Capability 제한** | 프로세스에 과도한 Root 권한을 주는 대신, `CAP_NET_ADMIN`이나 `CAP_SYS_RAWIO` 등 실행에 반드시 필요한 최소한의 리눅스 Capability만 부여해야 함. |
| **SYS-27** | 파일 시스템 및 네트워크 스택 보안 | **Sysctl 커널 파라미터 최적화** | `/etc/sysctl.conf` 설정을 통해 IP Forwarding 차단, ICMP Redirect 무시, SYN Flood 방어 등 네트워크 스택의 보안 설정을 강화해야 함. |
| **SYS-28** | 파일 시스템 및 네트워크 스택 보안 | **USB/외부 매체 차단** | `modprobe` 설정을 통해 `usb-storage`, `uas` 등의 모듈 로딩을 금지하여 USB를 통한 물리적 악성코드 유입을 차단해야 함. |
| **SYS-29** | 파일 시스템 및 네트워크 스택 보안 | **SSH Hardening** | 원격 접속 필요 시, 비밀번호 로그인을 금지하고 SSH 키 기반 인증만 허용하며, 루트 로그인 금지 및 비표준 포트 사용을 권장함. |
| **SYS-30** | 무결성 검증 | **IMA/EVM 활용** | 커널 수준의 무결성 측정 아키텍처(IMA)를 사용하여 실행 파일이나 설정 파일이 런타임 중 변조되었는지 실시간으로 검증해야 함. |
| **SYS-31** | 무결성 검증 | **Secure PPA/Repository** | 모든 소프트웨어 패키지는 서명된 공식 저장소에서만 업데이트되어야 하며, 서명되지 않은 외부 패키지 설치를 금지해야 함. |
| **AUD-01** | 시스템 및 보안 이벤트 로깅 | **핵심 시스템 콜 감시** | `Linux Audit Framework (auditd)`를 사용하여 `execve` (프로세스 실행), `ptrace` (디버깅), `setuid` (권한 상승) 등 보안 민감 시스템 콜을 실시간 기록해야 함. |
| **AUD-02** | 시스템 및 보안 이벤트 로깅 | **권한 상승 및 인증 로그** | `sudo` 사용 이력, SSH 로그인 시도, `/etc/shadow` 등 핵심 설정 파일에 대한 접근 및 수정 시도를 모두 로깅해야 함. |
| **AUD-03** | 시스템 및 보안 이벤트 로깅 | **리소스 남용 로깅** | CPU/메모리 사용량이 임계치를 초과하거나 특정 프로세스가 비정상적으로 자원을 독점하는 이벤트를 기록하여 DoS 공격 징후를 탐지해야 함. |
| **AUD-04** | 로그 무결성 및 보호 | **Append-Only 로그 구성** | 로컬 로그 파일은 '추가만 가능(Append-Only)' 하도록 설정(`chattr +a`)하여, 기존 기록을 수정하거나 삭제할 수 없도록 보호해야 함. |
| **AUD-05** | 로그 무결성 및 보호 | **TEE 기반 로그 해시 체인** | 로그의 각 블록을 하드웨어 보안 영역(TEE) 내의 키로 해싱하여 체인화함으로써, 로그의 일부가 누락되거나 순서가 바뀌는 것을 즉시 감지해야 함. |
| **AUD-06** | 로그 무결성 및 보호 | **원격 실시간 로그 전송** | 로컬 로그가 삭제될 경우에 대비하여, 생성된 로그를 즉시 클라우드 SIEM 또는 외부 보안 서버로 암호화 전송(mTLS 기반)해야 함. |
| **AUD-07** | 로봇 운영 및 제어 추적 | **Command-to-Action 매핑** | 클라우드 VLM에서 수신된 특정 명령 ID와 로봇이 실제 수행한 모터 제어 동작 간의 상관관계를 타임스탬프 기반으로 정밀 기록해야 함. |
| **AUD-08** | 로봇 운영 및 제어 추적 | **센서 데이터 오리진 검증** | 로봇이 클라우드로 보낸 영상/음성 데이터가 실제 물리 센서에서 발생한 것임을 증명하는 서명 정보를 로그와 함께 보관해야 함. |
| **AUD-09** | 로봇 운영 및 제어 추적 | **비상 정지(E-Stop) 이력** | 하드웨어 비상 정지 버튼 작동 또는 소프트웨어 Fail-Safe 트리거 이력을 기록하고, 당시의 시스템 상태 스냅샷을 보관해야 함. |
| **AUD-10** | 실시간 위협 탐지 및 알림 | **비정상 패턴 탐지** | 평소와 다른 시간대의 SSH 접속, 반복적인 인증 실패, 비인가 IP로의 대량 데이터 전송 등을 실시간 탐지해야 함. |
| **AUD-11** | 실 실시간 위협 탐지 및 알림 | **보안 위반 실시간 알림** | 심각한 보안 위반(SYS-15의 케이스 개방, SYS-16의 키 파기 등) 감지 시, 5초 이내에 관리자에게 즉시 알림(Alert)을 전송해야 함. |
| **AUD-12** | 실시간 위협 탐지 및 알림 | **포렌식 스냅샷** | 침해 사고 의심 시, 휘발성 메모리 상태 및 실행 중인 프로세스 목록을 덤프하여 사후 분석을 위한 데이터로 확보해야 함. |
