# 하드웨어 휴머노이드 특화 보안 요구사항

본 문서는 휴머노이드 로봇의 이기종(x86-NVIDIA) 보드 구조 및 물리적 특성을 고려한 하드웨어 레벨의 휴머노이드 특화 보안 요구사항을 정의합니다. 기존 KISA 보안 요구사항을 보완하며, 세부 휴머노이드 특화 보안 요구사항은 다음과 같습니다.

| ID | 분류 (Category) | 보안 요구사항 (Item) | 상세 요구사항 (Detailed Requirements) |
| :--- | :--- | :--- | :--- |
| **HDW-01** | 하드웨어 신뢰 및 무결성 | **Secure Boot** | UEFI(x86) 및 MB1/MB2(NVIDIA)에서 커스텀 키 기반의 전 단계 서명 검증을 강제해야 함. |
| **HDW-02** | 하드웨어 신뢰 및 무결성 | **Anti-Rollback** | 하드웨어 퓨즈(eFuse) 또는 보안 카운터를 활용하여 취약한 하위 버전 펌웨어 실행을 물리적으로 차단해야 함. |
| **HDW-03** | 하드웨어 신뢰 및 무결성 | **TEE/TPM** | 모든 개인키는 TEE(OP-TEE) 또는 TPM 2.0 내부에 생성 및 저장되어야 하며, 절대 메모리 밖(Normal World)으로 유출되지 않아야 함. |
| **HDW-04** | 하드웨어 신뢰 및 무결성 | **PUF (Physical Unclonable Function)** | 칩 고유의 미세한 물리적 특성(PUF)을 활용하여 장치 고유 ID를 생성하고, 이를 인증서 발급의 근거로 사용하여 하드웨어 복제를 방지해야 함. |
| **HDW-05** | 물리적 접근 통제 및 격리 | **JTAG/UART 비활성화** | 양산 및 파트너사 전달 버전에서는 하드웨어 디버깅 포트를 물리적/논리적으로 비활성화(Fuse Blowing)해야 함. |
| **HDW-06** | 물리적 접근 통제 및 격리 | **Memory Isolation** | IOMMU를 활성화하여 장치 드라이버가 허용되지 않은 메모리 영역(AI 모델 및 제어 로직 간)에 접근하는 것을 하드웨어 레벨에서 격리해야 함. |
| **HDW-07** | 물리적 접근 통제 및 격리 | **Physical Intrusion** | 로봇 본체 케이스 내부에 조도 센서나 기계적 스위치를 두어, 무단 개방 감지 시 TEE 내의 암호화 키를 즉시 파기(Self-Destruction)하도록 설계해야 함. |
| **HDW-08** | 물리적 접근 통제 및 격리 | **Fail-over Sync** | NVIDIA와 x86 사이의 이더넷 링크 상태를 1ms 단위로 감시하고, 물리적 단절(단선 등) 시 하드웨어 브레이크를 즉시 체결하는 독립적인 워치독 회로를 구성해야 함. |
